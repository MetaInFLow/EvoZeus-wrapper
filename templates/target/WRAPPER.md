## Runtime Status Check

1. Check the current Skill release against GitHub latest release.
2. Check `.evozeus_evoinfra/wrapper.json` against the current EvoZeus-wrapper harness version.
3. Check the source contract with `doctor`.
4. Continue to the target Skill's main flow only after all three are OK.

## Runtime Integration Mode

`.evozeus_evoinfra/wrapper.json` records `integration.mode`.

- `native_host_hook`: host/plugin lifecycle hooks are installed and evidenced by hook files plus plugin manifests.
- `bootstrap_skill`: plugin skill infrastructure exists, but no host lifecycle hook files were detected.
- `prompt_runtime_check`: the instruction surface asks the agent to run checks; this is prompt-compliance fallback.
- `manual_only`: wrapper commands must be run manually.

Wrapper CLI commands such as `hook start-check` are not runtime hooks unless a host integration calls them automatically. Do not describe this Skill as hook-backed unless `integration.mode` is `native_host_hook`.

## Evolution Loop

1. The target repo is the canonical Skill source. EvoZeus and runtime installs should point to it instead of keeping copied installs.
2. Before evolving or running a wrapper-managed Skill, resolve the source in this order:
   - read `.evozeus_evoinfra/wrapper.json`;
   - check `~/.evozeus/.projects/{{REPO_NAME}}`;
   - verify the canonical repo git origin / GitHub repo;
   - inspect runtime installs only as pointers;
   - use GitHub user/org/public search only as fallback when wrapper state is absent.
3. Before running the Skill, check whether GitHub has a newer release:
   `python3 scripts/evozeus_wrapper_preflight.py version --repo {{REPO_NAME}}`
4. User reports an unsatisfactory Skill result with the feedback Issue template.
5. Maintainer writes a design doc in `docs/designs/`.
6. Maintainer opens a PR referencing the design doc.
7. PR updates `SKILL.md` and `CHANGELOG.md`.
8. Release tag and release description are created after merge.

## Wrapper Harness Upgrade

The Skill release version and wrapper harness version are separate axes.

- Skill release is tracked by GitHub releases and `CHANGELOG.md`.
- Wrapper harness version is tracked by `.evozeus_evoinfra/wrapper.json`.
- Wrapper migrations are recorded under `docs/wrapper-migrations/`.

When EvoZeus-wrapper publishes a newer harness version, do not rewrite target Skill business rules. Generate an append-only migration plan from the EvoZeus-wrapper repo:

```bash
python3 scripts/evozeus_wrapper.py harness upgrade-check --target /absolute/path/to/this-skill --latest-version <wrapper-version> --json
python3 scripts/evozeus_wrapper.py harness upgrade --target /absolute/path/to/this-skill --latest-version <wrapper-version> --dry-run --json
```

The migration plan must state the from/to wrapper version, planned files, `SKILL.md` status-check action, append-only action, validation commands, and rollback plan.

## Release Version Standard

Use `vMAJOR.MINOR.PATCH`.

- `MAJOR`: incompatible Skill behavior or output contract change.
- `MINOR`: new capability, required evidence rule, or harness behavior.
- `PATCH`: wording, examples, bug fixes, validation fixes, or non-breaking clarifications.
