## Runtime Status Check

This repo is the maintainer artifact. Runtime-copy installs may contain only the target Skill instruction surface and referenced runtime files.

If this is a maintainer/canonical repo session, run wrapper status checks before changing, releasing, or migrating the Skill:

1. Check the current Skill release against GitHub latest release.
2. Check `.evozeus/wrapper.json` against the current EvoZeus-wrapper harness version.
3. Check the source contract with `doctor`.

If `.evozeus/wrapper.json` or wrapper tooling is unavailable, treat this as a runtime-only install and continue with the target Skill business flow. Do not block normal Skill execution only because wrapper governance files are absent.

For runtime integrations, prefer a start hook over manual prompt compliance:

```bash
python3 scripts/evozeus_wrapper.py hook start-check \
  --target /absolute/path/to/this-skill \
  --latest-version <latest-wrapper-version> \
  --json
```

The hook should continue only when `decision.allow` is true. A `warn` result may continue in advisory mode; a `block` result must stop wrapper-managed maintenance or strict hook execution.

## Evolution Loop

1. The target repo is the canonical Skill source. EvoZeus and runtime installs should point to it instead of keeping copied installs.
2. Before evolving, releasing, or migrating a wrapper-managed Skill, resolve the source in this order:
   - read `.evozeus/wrapper.json`;
   - check `~/.evozeus/.projects/{{REPO_NAME}}`;
   - verify the canonical repo git origin / GitHub repo;
   - inspect runtime installs only as pointers;
   - use GitHub user/org/public search only as fallback when wrapper state is absent.
3. Before maintainer-mode changes, check whether GitHub has a newer release:
   `python3 scripts/evozeus_wrapper_preflight.py version --repo {{REPO_NAME}}`
4. At turn end, run feedback audit when the user corrected the agent, expressed dissatisfaction, or requested a reusable behavior change:
   `python3 scripts/evozeus_wrapper.py loop audit --target /absolute/path/to/this-skill --user-input "<input>" --json`
5. If audit says capture is required, submit a feedback Issue according to `.evozeus/feedback-policy.json`.
6. Maintainer writes a design doc in `docs/designs/`.
7. Maintainer opens a PR referencing the design doc.
8. PR updates `SKILL.md` and `CHANGELOG.md`.
9. Release tag and release description are created after merge.

## Wrapper Harness Upgrade

The Skill release version and wrapper harness version are separate axes.

- Skill release is tracked by GitHub releases and `CHANGELOG.md`.
- Wrapper harness version is tracked by `.evozeus/wrapper.json`.
- Wrapper migrations are recorded under `docs/wrapper-migrations/`.
- Wrapper harness migrations that change the installable Skill artifact must also produce a target Skill patch release, even when target business rules do not change.

When EvoZeus-wrapper publishes a newer harness version, do not rewrite target Skill business rules. Generate an append-only migration plan from the EvoZeus-wrapper repo:

```bash
python3 scripts/evozeus_wrapper.py harness upgrade-check --target /absolute/path/to/this-skill --latest-version <wrapper-version> --json
python3 scripts/evozeus_wrapper.py harness upgrade --target /absolute/path/to/this-skill --latest-version <wrapper-version> --dry-run --json
```

The migration plan must state the from/to wrapper version, planned files, `SKILL.md` status-check action, append-only action, Skill patch release action, validation commands, and rollback plan.

## Release Version Standard

Use `vMAJOR.MINOR.PATCH`.

- `MAJOR`: incompatible Skill behavior or output contract change.
- `MINOR`: new capability, required evidence rule, or harness behavior.
- `PATCH`: wording, examples, bug fixes, validation fixes, or non-breaking clarifications.
